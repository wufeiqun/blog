## 临时解决

咨询了一下AI, 结论是主库上创建成功了, 从库上建表失败了, 失败的原因是该表的总长度超过了允许的上限, 临时解决的思路是放宽限制:

```
STOP SLAVE;

-- 关闭 strict_mode，模拟主库当时手动建表时绕过 1118 的方式
SET SESSION innodb_strict_mode = OFF;

-- 挂到卡住的那个 GTID 上（根据你最早 SHOW SLAVE STATUS 里 Executed_Gtid_Set 到 256751，下一个是 256752）
SET GTID_NEXT = 'b71ff515-4983-11ef-904f-ee70487f3ed2:256752';

USE aibusi;
-- 粘贴那条完整的 CREATE TABLE 语句, 可以在主库上执行 show create table查看

SET GTID_NEXT = 'AUTOMATIC';

START SLAVE;

```

后来成功了, 因为是双主, 此时主库上又报相同的错误了, 但是因为主库上已经创建了该表, 所以跳过该主从的事务就行:

```
STOP SLAVE;

-- 从 SHOW SLAVE STATUS 里的 Retrieved_Gtid_Set 或报错信息确认卡住的具体 GTID
-- 比如报错的是 b71ff515-4983-11ef-904f-ee70487f3ed2:256752
SET GTID_NEXT = 'b279773e-4983-11ef-90f3-ca43f0e5ade8:135';
BEGIN;
COMMIT;
SET GTID_NEXT = 'AUTOMATIC';

START SLAVE;
```

## 根因分析

这个其实是一个设计问题, 不应该让表搞的这么大, 应该拆分.

> 把最常用的核心字段留在一张“主表”里，把不常用的、或者是专门用来存大段文字的字段，放到另一张“扩展表”里，两张表通过主键关联起来。
