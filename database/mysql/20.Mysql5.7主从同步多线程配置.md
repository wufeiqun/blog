## MySQL5.7主从同步多线程配置

Mysql5.7默认的主从同步是单线程的, 如果遇到主库同时写入大量数据的时候, 从库就会出现同步延迟, 延迟的时间过长的话, 就会导致主从同步中断.

1. 在线配置

```
STOP SLAVE;
SET GLOBAL slave_parallel_workers = 8;
SET GLOBAL slave_parallel_type = 'LOGICAL_CLOCK';
SET GLOBAL innodb_flush_log_at_trx_commit = 2;
SET GLOBAL sync_binlog = 0;
START SLAVE;


```

参数说明:

```
slave_parallel_workers: 从库上用于处理主从同步的线程数, 默认是1个, 可以根据CPU数量调整, 比如4/8/16等.
slave_parallel_type: 并行拆分的逻辑, 默认是DATABASE, 意思是主库上只有两个作用于不同数据库上的事务才能在从库上并行处理, 这个比较慢, 因为大部分是同一个数据库, LOGICAL_CLOCK表示逻辑时钟, 意思是在主库上可以同时提交的两个事务在从库上就可以分配给不同的worker并行处理, 这个提高并行处理的速度.
innodb_flush_log_at_trx_commit: 事务日志落盘的频率, 默认是每次事务都会落盘, 最安全但是比较慢, 2是每次事务先写入OS, 由OS决定落盘方式
sync_binlog: binlog落盘方式, 默认1也是最安全的方式, 每次提交事务都会刷盘, 0表示由OS决定何时落盘.
```

2. 持久化配置(暂未详细研究)

每次临时调整即可, 没有使用持久化的方式.

```
[mysqld]
server-id = 2
relay_log = relay-bin
read_only = 1
log_slave_updates = 1

# 多线程复制配置
slave_parallel_type = LOGICAL_CLOCK
slave_parallel_workers = 8
master_info_repository = TABLE
relay_log_info_repository = TABLE
relay_log_recovery = ON
gtid_mode = ON
enforce_gtid_consistency = ON
binlog_format = ROW
```

可以使用`show processlist;`验证, 可以看到多个主从同步的线程.
